# 最適化監査レポート（多エージェント・実測付き）

PySide6/Qt製 CSV/波形 グラフ・オシロ解析アプリの最適化候補を、6観点で実測しながら洗い出し、優先度づけしたもの。計測は `C:\.venv` のPython・GUIは offscreen。

## 総括

6監査・全37指摘を統合し26項目に重複排除した。最大の論点は data_loader の engine='python' 固定で、読込・バッチ・メモリ・コード構造の4監査が独立に指摘していたため rank1 に集約（C エンジン化で7倍速・float32 で DataFrame 半減、small/low）。同様に _setup_dynamic_resample の重複数値化（描画監査）と dyn の40MB常駐コピー（メモリ監査）は同根のため rank11 に、描画オプション16式重複と series構築重複は DRY 系として rank10/20 に統合した。優先順位は impact高×effort小×risk低を最上位とし、起動・読込・描画・バッチの即効4本（rank1-3,6）＋メモリリーク修正（rank7）＋GUIの差分更新（rank4-5）を上位に置いた。これらは合計で起動約0.84s短縮・500kファイル読込1.5s→0.25s・間引き33.9倍・style表 N=1000 で8秒→ほぼ0・40MBリーク解消を、いずれも小工数・低リスクで達成できる。effort大の本格対応（ライブ更新高速パス rank8、batch並列化 rank9、god class 分割 rank18）は実測効果が大きいものの設計変更を伴うため即効群の後に配置した。なお rank11/17/25 と rank4/5/13/14/26 はそれぞれ実装が重複するため統合着手を推奨する。

## 観点別の詳細findings（実測根拠つき）

### 起動・読み込みパフォーマンス（GraphApp初期化 / _load_file / data_loader / _refresh_columns / 多数ファイル読込）

**load_table が engine="python" 固定で C エンジンの約7倍遅い（最大の読み込みボトルネック）**  
`impact=high / effort=small / risk=medium`  
- 根拠: data_loader.py:147 `read_kwargs = dict(sep=delimiter, engine="python", ...)`。実測(500k点・34MBの 波形_負荷テスト_500k.csv、C:/.venv): load_table 全体=1486ms。pd.read_csv 同条件で engine='c'=204ms(best of 3) vs engine='python'=1595ms(best of 2)。差は約1.3秒/ファイル。_load_file はメインスレッド同期実行(graph_app.py:1036)のため、この時間ぶんUIがフリーズする。
- 対応: 区切り文字が単一文字(','、'\t'、';'、'|')の通常ケースでは engine='c' を使う。まず engine='c' で読み、ParserError 等で失敗したときだけ engine='python' にフォールバックする2段構えにする。これだけで500kファイルの読込が約1.5秒→約0.25秒に短縮見込み。複数文字sepや特殊ケースのみ python エンジンを残せば機能は維持できる。

**起動時に scipy.signal がモジュールロードされ、import コストの約半分(約1.1秒)を占める**  
`impact=high / effort=small / risk=low`  
- 根拠: `python -X importtime` 累積プロファイル: graph_app 全体=2104ms のうち scipy.signal=1097ms(scipy.signal._support_alternative_backends=1059ms 等)。連鎖は graph_app→advanced→analysis、analysis.py:13-14 のモジュール先頭 `from scipy.signal import find_peaks`(_HAVE_SCIPY フラグ用)が原因。実際の find_peaks/savgol/flattop 等の使用箇所(analysis.py:45,76,208,576)は既に関数内で遅延importしている。A/B実測: analysis.py の先頭importを遅延化すると cold `import graph_app` が約1770ms→約930ms(3回平均、scipy.signal未ロード)に短縮＝約840ms(47%)削減。
- 対応: analysis.py:13-17 のモジュール先頭 `from scipy.signal import find_peaks` を削除し、_HAVE_SCIPY を遅延判定にする。例: find_signal_peaks 内で `try: from scipy.signal import find_peaks; ... except ImportError: 簡易ピーク` とするか、初回呼び出し時にimportする `_get_find_peaks()` ヘルパへ。解析を一度も実行しないユーザー(=単純な描画用途)は scipy ロード約1.1秒を完全に回避できる。

**detect_encoding がファイル全体をバイナリ読み込み＋全体を複数回decodeしている**  
`impact=medium / effort=small / risk=medium`  
- 根拠: data_loader.py:46-47 `raw = f.read()`(全34MB読込)、続く data_loader.py:56 `raw.decode("utf-8")` が全体を再デコード(失敗時は cp932/euc-jp も全体decode)。実測(500kファイル): detect_encoding=16-20ms、tracemalloc 追跡ピーク=34.0MB(ファイルサイズ相当)。先頭64KBのみ読む試作版は実測 0.43ms / ピーク0.074MB。BOM/UTF-8判定は先頭サンプルで十分。

### パフォーマンス / 一括出力（batch_export）の最適化

**batch_export の逐次ループを multiprocessing 並列化する（最大の効果）**
